## "Primary keys all are real" — how correctness is guaranteed

PKs are never sampled randomly: `int` PKs are sequences (1..N), `uuid` PKs are
RFC-4122 v4, business PKs use `format` templates ("CUST-######"). All are
generated by construction-unique samplers, then re-verified by the quality
engine's uniqueness check and the FK inclusion check. Person/contact values
(names, emails, phones, cities) come from dedicated realistic samplers keyed on
column names — and the propose_spec system prompt now explicitly forbids the AI
from overriding them with gibberish `format` patterns (fix from live review).

## Architecture alignment

The proposed pipeline (Prompt → Research → Spec → Model → Rules → Generate →
Validate → Output) is exactly the implemented flow:

```
user prompt → research_and_generate (client web search, cited)
           → propose_spec (LLM reasoning → validated YAML)
           → apply_spec (catalog + joins + rules)
           → generate_synthetic_data (Plan IR, FK-safe, seeded)
           → run_quality_check (explained score) → preview_data
           → output: csv / parquet / sql / duckdb
```

## Recommended next 3 (in order of value)

1. **examples/scenarios/** — curated spec templates for banking-KYC, healthcare-claims, churn, fraud, pump-failure (pure YAML, zero code — this IS the industry-rules pack).
2. **Seasonality spec key** (`monthly_weights:` on date columns) — the one realism pattern not yet declarative.
3. **DDL/SQL-file connector** — apply a CREATE TABLE script as schema input (closes "analyze user-provided schema" fully).
